# ReportFindings

코드 리뷰 발견사항을 호스트 UI가 기본적으로 렌더링하는 타입화되고 구조화된 목록으로 보고합니다 — 채팅 텍스트로 인쇄하지 않습니다.

## 사용 시점

- 활동 지침이 명시적으로 이 도구로 발견사항을 보고하도록 말하는 코드 리뷰를 결론짓을 때
- 수정을 적용한 후 재보고할 때, 리뷰의 적용 지침이 이를 요청할 때 (각 발견사항은 그러면 `outcome`을 포함합니다)
- **하지 않을 때**: 임시 의견, 일반 답변, 또는 리뷰 지침이 다른 출력 형식을 지정하는 경우 — 절대로 동일한 발견사항의 텍스트 중복과 함께 사용하지 마세요

## 매개변수

- `findings` (배열, 필수, 최대 32개): 검증된 발견사항, 가장 심각한 것부터 순위 지정됨 — 검증을 통과한 발견사항이 없으면 빈 배열. 각 발견사항:
  - `file` (문자열, 필수): 저장소 상대 경로.
  - `line` (숫자, 선택사항): 1-인덱싱 앵커 라인.
  - `summary` (문자열, 필수): 결함의 한 문장 진술.
  - `failure_scenario` (문자열, 필수): 구체적인 입력/상태 → 잘못된 출력 또는 크래시.
  - `category` (문자열, 선택사항): 짧은 kebab-case slug, 예: `correctness`, `simplification`, `efficiency`, `test-coverage`.
  - `verdict` (문자열, 선택사항): `CONFIRMED` 또는 `PLAUSIBLE` — 검증 통과 시 설정; 인라인 전용 리뷰에서는 없음.
  - `outcome` (문자열, 선택사항): 수정 후 재보고할 때만 — `fixed`, `skipped` 또는 `no_change_needed`.
- `level` (문자열, 선택사항): 리뷰가 실행된 노력 수준 — `low`, `medium`, `high`, `xhigh` 또는 `max`.

## 참고사항

- **한 번 호출합니다.** 완전하고, 검증되고, 심각도 순위 지정된 목록이 있는 단일 호출 — 발견사항당 하나의 호출이 아닙니다.
- **비어있음은 유효한 결과입니다.** 검증을 통과한 발견사항이 없으면 약한 발견사항으로 채우지 말고 빈 배열을 보고합니다.
- **텍스트에서 중복하지 마세요.** 이 도구가 결과를 보고할 때 발견사항을 채팅 메시지로도 인쇄하면 안 됩니다.
- **`outcome`은 재보고 전용입니다.** 첫 번째 보고에서는 설정하지 않습니다; 적용 통과 후 각 발견사항에 실제로 일어난 것을 설정합니다.
