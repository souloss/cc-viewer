# ReportFindings

将代码审查发现报告为类型化、结构化列表，由宿主 UI 原生呈现——而不是将其打印为聊天文本。

## 何时使用

- 结束代码审查，其活动指令明确说明使用此工具报告发现
- 应用修复后重新报告，当审查的应用指令要求此操作时（每个发现随后携带 `outcome`）
- **不适用于**临时意见、普通答案或审查指令指定不同输出格式的情况——绝不与同一发现的文本副本一起使用

## 参数

- `findings`（数组，必填，最多 32 个）：已验证的发现，按最严重优先排序——如果没有发现通过验证，则为空数组。每个发现：
  - `file`（字符串，必填）：相对于仓库的路径。
  - `line`（数字，可选）：1 索引锚点行。
  - `summary`（字符串，必填）：缺陷的单句陈述。
  - `failure_scenario`（字符串，必填）：具体输入/状态 → 错误输出或崩溃。
  - `category`（字符串，可选）：短 kebab-case slug，例如 `correctness`、`simplification`、`efficiency`、`test-coverage`。
  - `verdict`（字符串，可选）：`CONFIRMED` 或 `PLAUSIBLE`——验证通过时设置；内联审查中不存在。
  - `outcome`（字符串，可选）：仅在修复后重新报告时——`fixed`、`skipped` 或 `no_change_needed`。
- `level`（字符串，可选）：审查运行时的工作量级别——`low`、`medium`、`high`、`xhigh` 或 `max`。

## 注意事项

- **调用一次。** 一次调用包含完整、已验证、按严重程度排序的列表——而不是每个发现调用一次。
- **空是有效结果。** 如果没有发现通过验证，报告空数组而不是用弱发现补充。
- **不要在文本中重复。** 当此工具报告结果时，发现不得也打印为聊天消息。
- **`outcome` 仅用于重新报告。** 在第一次报告时不设置它；应用通过后，设置每个发现实际发生的情况。
