# ReportFindings

รายงานสิ่งที่ค้นพบจากการ code-review เป็นรายการที่พิมพ์และมีโครงสร้าง ซึ่ง UI โฮสต์แสดงผลเป็นดั้งเดิม — แทนที่จะพิมพ์เป็นข้อความแชท

## เมื่อใดควรใช้

- บสรุปการ code-review ซึ่งคำแนะนำที่ใช้งานจริงระบุอย่างชัดเจนให้รายงานสิ่งที่ค้นพบด้วยเครื่องมือนี้
- รายงานอีกครั้งหลังจากใช้การแก้ไข, เมื่อคำแนะนำการใช้ของการ review ขอให้ (แต่ละสิ่งที่ค้นพบจึงมี `outcome`)
- **ไม่ใช้** สำหรับความเห็นแบบ ad-hoc, คำตอบธรรมชาติ, หรือการ review ซึ่งคำแนะนำระบุรูปแบบผลลัพธ์ที่แตกต่าง — และไม่เคยควบคู่ไปกับสำเนาข้อความของสิ่งที่ค้นพบเดียวกัน

## พารามิเตอร์

- `findings` (array, จำเป็น, สูงสุด 32): สิ่งที่ค้นพบซึ่งตรวจสอบแล้ว, จัดลำดับที่ร้ายแรงที่สุดก่อน — อาร์เรย์ว่างถ้าไม่มีสิ่งใดที่รอดการตรวจสอบ แต่ละสิ่งที่ค้นพบ:
  - `file` (สตริง, จำเป็น): เส้นทางสัมพัทธ์ repo
  - `line` (number, ไม่จำเป็น): บรรทัดที่ใช้ยึดตำแหน่ง เริ่มนับจาก 1
  - `summary` (สตริง, จำเป็น): การระบุข้อบกพร่องประโยคเดียว
  - `failure_scenario` (สตริง, จำเป็น): inputs/state ที่เป็นรูปธรรม → output ที่ผิดหรือ crash
  - `category` (สตริง, ไม่จำเป็น): slug kebab-case สั้น, เช่น `correctness`, `simplification`, `efficiency`, `test-coverage`
  - `verdict` (สตริง, ไม่จำเป็น): `CONFIRMED` หรือ `PLAUSIBLE` — ตั้งเมื่อการ verify pass ทำงาน; ขาดใน inline-only reviews
  - `outcome` (สตริง, ไม่จำเป็น): เฉพาะเมื่อรายงานอีกครั้งหลังจากการแก้ไข — `fixed`, `skipped`, หรือ `no_change_needed`
- `level` (สตริง, ไม่จำเป็น): ระดับความพยายามที่ review ทำงาน — `low`, `medium`, `high`, `xhigh`, หรือ `max`

## หมายเหตุ

- **เรียกครั้งเดียว** การเรียกเดียวกับรายการที่สมบูรณ์ ตรวจสอบแล้ว, จัดลำดับความรุนแรง — ไม่ใช่การเรียกต่อสิ่งที่ค้นพบ
- **ว่างเปล่าคือผลที่ถูกต้อง** หากไม่มีสิ่งที่ค้นพบซึ่งรอดการตรวจสอบ, รายงานอาร์เรย์ว่างแทนการเติมเต็มด้วยสิ่งที่ค้นพบที่อ่อนแอ
- **อย่าทำซ้ำในข้อความ** เมื่อเครื่องมือนี้รายงานผลลัพธ์, สิ่งที่ค้นพบจะไม่ถูกพิมพ์เป็นข้อความแชทด้วย
- **`outcome` เป็นเพียงสำหรับการรายงานอีกครั้ง** ในรายงานแรกให้ไม่ตั้ง; หลังจากการ apply pass, ตั้งว่าจริง ๆ เกิดขึ้นสำหรับแต่ละสิ่งที่ค้นพบ
